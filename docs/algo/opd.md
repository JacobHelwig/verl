# On-Policy Distillation (OPD)

Last updated: 03/22/2026.

On-policy distillation (OPD) trains a student model on samples generated by the student's current policy, while a stronger teacher provides token-level supervision for those same trajectories. In verl, this lets you reuse the PPO-style training loop and rollout stack while replacing or augmenting task rewards with distillation signals.

The implementation added in PR `455e44c6feeabe378ad9dcc790fe9313926de12c` extends verl's main training path with first-class OPD support across both FSDP and Megatron backends. The core design is:

- The student model generates on-policy rollouts with the existing rollout engine.
- A teacher model computes log-probability targets for the student's prompt and sampled response.
- The trainer converts those teacher targets into either a supervised distillation loss or a policy-gradient reward signal.
- The actor update then runs through the same `main_ppo` entrypoint used by other verl algorithms.

This implementation supports:

- FSDP and Megatron training backends
- LLM and VLM distillation
- Top-k forward-KL distillation and KL-estimator-based reverse-KL losses
- Supervised updates and policy-gradient-style updates
- Teacher inference with a vLLM-backed teacher model
- FSDP sequence parallel, and Megatron tensor/context parallel

For asynchronous scheduling recipes built on top of distillation, see [Async On-Policy Knowledge Distillation Trainer](../advance/async-on-policy-distill.md).

## OPD Key Components

- On-policy teacher scoring: after the student rollout is generated, verl sends the sampled prompt and response to a teacher worker that computes log probabilities on the exact sampled tokens.
- Two distillation families:
  - `forward_kl_topk`: forward KL on the teacher's top-k support
  - KL estimators such as `k1` and `k3`: reverse-KL-style estimators computed from sampled-token log probabilities
- Two update modes:
  - Supervised distillation: directly backpropagates the distillation loss
  - Policy-gradient distillation: uses the negative distillation loss as a reward signal
- Shared trainer path: OPD is enabled through the standard `verl.trainer.main_ppo` flow rather than a separate trainer binary.
- Multi-engine support: the same feature lands in both FSDP and Megatron actor implementations, with engine-specific loss kernels where needed.

One important design choice in the PR is how teacher log probabilities are computed. Instead of routing requests through an OpenAI-compatible text endpoint, the implementation adds a direct `compute_logprobs` path in the vLLM server stack. This preserves token-level alignment between the student's sampled IDs and the teacher's scoring input, which is especially important for exact distillation and multimodal support.

## OPD Configuration

Note that all configs containing `micro_batch_size` are used to configure the maximum sample or token count per forward or backward pass to avoid GPU OOMs, whose value should not change algorithmic/convergence behavior.

Despite many configs still living under `ppo_` trainer names, they also apply to OPD because the distillation implementation is integrated into the PPO training path.

- `distillation.enabled`: Enables OPD. Default is `False`.

- `distillation.num_workers`: Number of teacher workers used to parallelize teacher logprob computation. Default is `8`.

- `distillation.teacher_model.model_path`: Teacher checkpoint or Hugging Face model ID.

- `distillation.teacher_model.enable_resource_pool`: Whether the teacher gets a dedicated resource pool. This is useful when the teacher should run on separate GPUs, especially for larger teachers or VLM setups.

- `distillation.teacher_model.n_gpus_per_node`, `distillation.teacher_model.nnodes`: Teacher resource allocation when a separate teacher pool is enabled.

- `distillation.teacher_model.inference.name`: Teacher inference backend. The current OPD implementation expects a rollout-style inference engine and supports `vllm`.

- `distillation.distillation_loss.loss_mode`: Distillation loss to use. Common choices:
  - `forward_kl_topk` for top-k forward KL
  - `k1` for policy-gradient-style OPD
  - `k3` for KL-estimator-based supervised distillation

- `distillation.distillation_loss.topk`: Number of teacher logits kept for top-k losses. This matters for `forward_kl_topk`.

- `distillation.distillation_loss.use_policy_gradient`: Chooses between policy-gradient OPD and supervised OPD.

- `distillation.distillation_loss.use_task_rewards`: Whether to combine normal task rewards with the distillation signal. In the example scripts from this PR, it is set to `False` so the runs measure pure distillation behavior.

- `distillation.distillation_loss.distillation_loss_coef`: Scaling factor when distillation is combined with task rewards.

- `distillation.distillation_loss.loss_max_clamp`: Optional upper clamp for the distillation loss. The PR results found this useful for training stability.

- `distillation.distillation_loss.log_prob_min_clamp`: Optional lower clamp on log probabilities for numerical stability.

- `algorithm.adv_estimator=grpo`: The example scripts use the GRPO advantage path together with OPD.

- `algorithm.use_kl_in_reward=False`: Avoids stacking the standard policy-reference KL reward on top of the distillation signal.

For stability, the current implementation enforces an important constraint:

- `k1` should be used with `use_policy_gradient=True`
- `forward_kl_topk` is most effective with `use_policy_gradient=False`

## Recommended Setups

### Supervised OPD

Use this when you want the teacher signal to be directly backpropagated as a distillation loss.

- `distillation.distillation_loss.use_policy_gradient=False`
- `distillation.distillation_loss.loss_mode=forward_kl_topk` or `k3`

This is the recommended starting point for standard LLM distillation.

### Policy-Gradient OPD

Use this when you want the negative distillation loss to act as a reward.

- `distillation.distillation_loss.use_policy_gradient=True`
- `distillation.distillation_loss.loss_mode=k1`

This is the setup used by the VLM example in the PR, and it matches the "distillation-as-reward" formulation described in the Thinking Machines writeup.

## Practical Notes

- Teacher context length must be large enough to score the full student prompt, the full sampled response, and one extra token.
- If you use top-k distillation, the teacher backend must expose at least `topk` log probabilities.
- The PR's experiments found top-k forward KL benefits from clamping for stability.
- The PR also found top-k reverse KL and top-k JSD variants to be unstable, so they are not the main recommended paths.
- If you need asynchronous overlap between rollout, teacher scoring, and updates, use the advanced async distillation recipe rather than this baseline algorithm page.

For more reference performance and related algorithm context, please see [baseline](https://verl.readthedocs.io/en/latest/algo/baseline.html) and [GRPO](https://verl.readthedocs.io/en/latest/algo/grpo.html).
