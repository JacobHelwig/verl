# On-Policy Distillation (OPD)

Last updated: 03/24/2026.

In this introduction, we motivate OPD and tell how it operates. 

```add
Why OPD? 
- Tradiional KD uses rollouts sampled from teacher to optimize student using next token prediction loss. 
- At inference time, there is misalginment with training samples, since rollouts are sampled from student. Student will reach states that are from a different distributoin than it saw during training. It has never been able to acquire knowledge from the teacher about how to handle these states.
- For example, suppose we are training for a coding task. Perhaps the teacher has a preference for solving graph traversal problems using BFS, but student prefers to use DFS. While all training data for graph traversals generated by the teacher will rely on BFS, at inference time, if the student decides to use DFS, it will not have acquired knowledge from the teacher about improving its DFS abilities
- To address this, OPD flips things around: instead of generating from the teacher, it generates rollouts from the student and then compares student and teacher distribution at each state to compute a divergence and use this divergence as the optimization signal
- in this way, sstudent has been able to acquire teacher knowledge for states that are aligned with what it will encounter during inference
```

OPD operates as:

- The student generates an on-policy rollout.
- Compute teacher logprobs for student generation.
- Use teacher logprobs and student logprobs to compute token-level divergences.
- Use these divergences with an optimization algo to update student to minimize divergence

That gives OPD two main design levers:


## Key Components 

- Student rollouts: same as in regular RL
- Online teacher: different from vanilla distillation, OPD requires online access to teacher to compute logprobs for student. We implement teacher as a teacher server; it lives on a vLLM/ sglang server and we send the student rollouts as prompts and compute the logprobs using it 
- Teacher from same model family: generic OPD requires student and teacher to be from same model family so that tokenizer are aligned and logprobs can be computed for same positions 
- Divergence function: requires us to choose a divergence. We support top-k forward KL and estimator reverse KL
  - `forward_kl_topk`: forward KL on the teacher's top-k support
  - KL estimators such as `k1` and `k3`: reverse-KL-style estimators computed from sampled-token log probabilities
- Optimization algo: given divergences, how to update student? We support supervised, ie backpropping directly on divergences and policy gradient, ie treating negative divergence as a reward to maximize.
  - Supervised distillation: directly backpropagates the distillation loss (GKD paper https://arxiv.org/abs/2306.13649)
  - Policy-gradient distillation: uses the negative distillation loss as a reward signal (Thinking Machinea https://thinkingmachines.ai/blog/on-policy-distillation/)
  - include both of these in references below

## Configuration 

The student model is configured in the same way as the policy trained by RL algos such as PPO.

Teacher model and other distillation settings are configured using `distillation` cfg

- `distillation`: distillation config
  - `enabled`: Enables OPD. Default is `False`.
  - `teacher_model`: config for teacher model
    - `model_path`: Teacher checkpoint or Hugging Face model ID.
    - `enable_resource_pool`: Whether the teacher gets a dedicated resource pool. If the teacher has its own resource pool (**standalone mode**), then it runs as part of the agent loop, scoring generations as the student finishes them. If the teacher shares the global resource pool with the student (**colocate mode**), then it scores student generations after the student has finished all generations and been offloaded.
    - `n_gpus_per_node`, `nnodes`: Teacher resource allocation when a separate teacher pool is enabled.
    - `inference.name`: Teacher inference backend, e.g. `vllm`.
  - `distillation_loss`: config for how teacher and student are compared
    - `loss_mode`: Distillation loss to use. Common choices:
      - `forward_kl_topk` for top-k forward KL
      - `k1`, `k3` for KL-estimator-based losses
    - `topk`: Number of teacher logits kept for top-k losses.
    - `use_policy_gradient`: Chooses between policy-gradient OPD and supervised OPD.
    - `use_task_rewards`: Whether to combine normal task rewards with the distillation signal. In the example scripts from this PR, it is set to `False` so the runs measure pure distillation behavior.
    - `distillation_loss_coef`: Scaling factor when distillation is combined with task rewards.
    - `loss_max_clamp`: Optional upper clamp for the distillation loss.
    - `log_prob_min_clamp`: Optional lower clamp on log probabilities for numerical stability.

## Understanding the OPD Framework

The OPD implementation in verl has four main stages:

1. The student generates responses with rollout engine.
2. The teacher computes logprobs for the student generation using a teacher inference server.
3. Student logprobs and teacher logprobs are compared using the chosen divergence metric to compute token-level divergences.
4. The actor update consumes those divergences either as a supervised objective or as a reward signal to update the student policy.


There are two main loss families:

- `forward_kl_topk`: uses teacher top-k logits and computes a sparse forward KL on that support.
- Estimator losses such as `k1` and `k3`: use only the sampled token log probability, similar to the KL estimators already used elsewhere in verl.

`forward_kl_topk` is the richer signal because it tells the student how to move probability mass over the teacher's top-k support, not just the sampled token. However, due to the need of passing around the teacher top-k probs between ray actors and gathering these with the student, this is more expensive in time and memory than estimator losses.

There are also two update styles:

- Supervised OPD: directly backpropagates the distillation loss through the actor.
- Policy-gradient OPD: treats negative distillation loss as reward and updates the actor with a policy-gradient loss.

Its important to note that using top-k loss with PG OPD is not a good idea, since PG only directly influences the direction of movement of the sampled token logprob (either up or down), while top-k loss includes additional signal as to the other logprobs in the top-k. In addition to wasting this information, the reward signal can be noisy: for example, the teacher logprob and student logprob for the sampled token could align perfectly, but all other tokens in top k might be completely different from one another, leading to a large divergence and large update of the sampled token.

Additionally, using the k1=p - q estimator with supervised update is not possible, since gradient of k1 loss wrt student weights does not depend on the teacher logprobs, and will instead lead to the an update that minimizes the probability of the samepled logprob

## OPD Metrics

All distillation-specific metrics are prefixed with `actor/distillation/` in logs. For example, the max distillation loss across all tokens in a batch appears as `actor/distillation/loss_max`.

### Core OPD Metrics

- `distillation/loss`: aggregated distillation loss when distillation is combined into the final actor objective.
- `distillation/loss_min`: minimum token-level distillation loss over valid response tokens.
- `distillation/loss_max`: maximum token-level distillation loss over valid response tokens.

### top-*k* Metrics

- `distillation/student_mass`: mean probability mass that the student assigns to the teacher's top-k support.
- `distillation/student_mass_min`
- `distillation/student_mass_max`
- `distillation/teacher_mass`: mean probability mass retained in the teacher's top-k truncation.
- `distillation/teacher_mass_min`
- `distillation/teacher_mass_max`

These are the first metrics to inspect when `forward_kl_topk` behaves strangely. If the captured mass is too low, then it can indicate instability or that the sparse top-k approximation may be discarding too much of the distribution.

### Estimator Metrics 

- `distillation/abs_loss`: mean absolute per-token distillation loss over valid response tokens.

This is useful the  `k1` can be negative at the token level, so raw signed means can hide how large the mismatch actually is.

### Additional metrics when `use_policy_gradient=True`

When distillation is routed through the policy-gradient path, verl also logs the policy-loss metrics under the `distillation/` prefix, including:

- `distillation/pg_clipfrac`
- `distillation/pg_clipfrac_lower`
- `distillation/ppo_kl`

These tell you how hard the policy-gradient update is pushing, and whether the distillation reward signal is causing aggressive clipping or large policy movement.

### General training metrics that remain useful

- response length metrics such as `response_seq_len/*`
- timing metrics such as `timing/*`
- throughput metrics such as `perf/*`
- validation reward / accuracy metrics when your run includes evaluation

## OPD Troubleshooting

- Teacher context length must be large enough to score the full student prompt, the full sampled response, and one extra token.
- If you use top-k distillation, the teacher backend must expose at least `topk` log probabilities.
- The PR's experiments found top-k forward KL benefits from clamping for stability.
- The PR also found top-k reverse KL and top-k JSD variants to be unstable, so they are not the main recommended paths.

If you are trying to add a new OPD variant and it is not working, the first debugging step should usually be:

- set `teacher_model == student_model`

That gives you a much simpler sanity check. If loss or training behavior still looks wrong in the self-distillation case, the issue is more likely in the plumbing than in the teacher-student capability gap.

For `forward_kl_topk`, inspect:

- `distillation/student_mass`
- `distillation/teacher_mass`

If the amount of mass captured in the top-k probabilities is too small, the loss may become noisy or misleading. In that case, increasing `topk` or checking whether the teacher and student distributions are unusually flat is often the right next step.

If things still are not working, it is often worth setting a breakpoint inside the loss and inspecting which tokens contribute the largest per-token losses. In practice, very large divergences often come from a small number of stylistically mismatched tokens, such as different answer formatting, different use of delimiters, or different response conventions between student and teacher.

If you need asynchronous overlap between rollout, teacher scoring, and updates, use the advanced async distillation recipe rather than this baseline algorithm page.

## Notes/Limitations
- limitation that we can't gather student top-k logprobs from teacher server
- limitation that top-k distillation using reverse/ KL divergences explored by agarwal might not be stable
